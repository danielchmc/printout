# Project Specification: Mixed-Format Printout PDF Tool

## 1. Project Summary

Build a tool that takes one or more PDF files generated by the existing shop-admin printout system and combines their individual price-sign regions into optimized A4 print sheets.

The goal is to solve the current limitation where different printout formats, such as A5, A6, A7, shelf labels, spear labels, in-ear labels, and over-ear labels, cannot be mixed efficiently on the same sheet of paper.

The proposed workaround does not require changes to the existing backend. Instead, it post-processes the PDFs that the backend already generates.

## 2. Core Problem

The current printout system can export several products at once, but only within one selected format per export. For example, it can generate multiple A7 labels on an A4 sheet, or multiple A5 labels on an A4 sheet, but it cannot generate a mixed sheet containing one A5 label, one A6 label, and several A7 labels.

In real shop usage, price updates often involve only three or four items, and those items often need different printout formats. This forces the user to export and print several separate PDFs, wasting paper and time.

## 3. Goal

Create a tool that can:

1. Accept existing backend-generated PDF printouts.
2. Detect the individual printable regions inside those PDFs.
3. Extract only the occupied label/sign regions.
4. Repack those regions onto as few A4 sheets as possible.
5. Export a new print-ready A4 PDF.

## 4. Non-Goals

The first version should not attempt to replace the existing shop-admin printout system.

It should not require modifying the existing backend PDF generator.

It should not rely on filenames to identify printout formats.

It should not try to redesign the labels or recreate them from product data.

It should not crop tightly around text or images, because the whitespace inside each label is part of the intended design.

## 5. Main Concept

The backend already produces printouts on A4 sheets. When a sheet contains multiple labels, the labels are separated by long straight black lines. These lines will be referred to as **cut-lines**.

The tool should detect cut-lines and use them to split a source PDF page into logical print regions.

Instead of asking, "Is this file A5, A6, or A7?", the tool primarily asks:

> "Where are the cut-lines, and what regions do they divide the page into?"

Each resulting region can then be inspected to determine whether it contains actual product content. Occupied regions are extracted and placed onto a new A4 sheet.

## 6. Input

The tool accepts one or more PDF files exported from the existing shop-admin system.

Typical inputs:

- One A5 product PDF
- One A6 product PDF
- One A7 product PDF
- One PDF containing multiple same-format products
- Custom format PDFs such as shelf, spear, in-ear, or over-ear labels

The tool must not assume that filenames contain useful information.

## 7. Output

The tool outputs one print-ready A4 PDF containing mixed-format labels packed onto as few pages as possible.

The output should preserve:

- visual quality,
- text sharpness where possible,
- original label proportions,
- official label layout,
- internal whitespace,
- cut boundaries,
- and print scale.

## 8. Detection Strategy

### 8.1 Render-Based Analysis

For each PDF page, render the page to an image at a suitable resolution, for example 150-300 DPI.

The rendered image is used for analysis only. The final PDF output should preferably use vector-preserving PDF operations where possible.

### 8.2 Cut-Line Detection

The tool should detect long, straight horizontal and vertical black or dark gray lines.

A detected line should only be accepted as a cut-line if it meets rules such as:

- It is mostly horizontal or mostly vertical.
- It is long relative to the page size.
- It is thin compared to the page size.
- It is continuous or nearly continuous.
- It is not just part of product text or logo graphics.
- It is located in a plausible splitting position.

Possible detection techniques:

- grayscale conversion,
- dark-pixel thresholding,
- morphological line detection,
- projection profiles along the x-axis and y-axis,
- Hough line transform,
- connected component analysis.

The initial implementation can use projection profiles and morphological operations because the expected cut-lines are simple straight lines.

### 8.3 Region Construction

Once cut-lines are detected, the page is split into rectangular regions.

Example:

- One vertical cut-line -> two regions.
- One vertical and one horizontal cut-line -> four regions.
- Multiple vertical and horizontal cut-lines -> grid of regions.

If no cut-lines are found, the entire page is treated as one region, unless fallback template detection identifies a known layout.

### 8.4 Occupancy Detection

Each region is analyzed to determine whether it contains product content.

The tool should ignore:

- the cut-line itself,
- page edges,
- tiny dust/noise pixels,
- insignificant artifacts.

A region is considered occupied if it contains enough non-white/non-background pixels after ignoring borders and cut-lines.

This prevents empty slots from being extracted.

### 8.5 Region Size Classification

Once a region is extracted, classify its physical size based on its dimensions.

Examples:

- Approximately full A4 -> A4
- Approximately half A4 -> A5
- Approximately quarter A4 -> A6
- Approximately one-eighth A4 -> A7

Custom sizes can be classified as custom regions, or mapped to known shop formats if their dimensions are stable.

## 9. Fallback Strategy

Cut-line detection should be the primary method, but not the only method.

If cut-line detection is uncertain, the tool should fall back to predefined layout templates.

Fallback templates can define known source layouts such as:

- A4 full page
- A5 two-up layout
- A6 four-up layout
- A7 eight-up layout
- shelf label layout
- spear label layout
- in-ear label layout
- over-ear label layout

The fallback should use:

- page size,
- orientation,
- detected content bounding box,
- approximate occupied area,
- known slot dimensions.

## 10. Extraction Rules

The tool must extract full label regions, not tightly crop around visible content.

Correct behavior:

> Detect an occupied A7 slot, then crop the full A7 slot rectangle.

Incorrect behavior:

> Crop tightly around the product name, price, and SKU.

This is important because the label's internal whitespace is part of the visual design and print layout.

## 11. Packing Rules

The extracted regions should be packed onto A4 pages.

The packing algorithm should prioritize:

1. Preserving exact scale.
2. Avoiding overlap.
3. Using as few A4 sheets as possible.
4. Keeping labels aligned to sensible cut positions.
5. Avoiding unnecessary rotation unless enabled.

### 11.1 Basic Packing

A simple first version can use a grid-based packing strategy:

- A4 labels occupy a full page.
- A5 labels occupy half a page.
- A6 labels occupy a quarter page.
- A7 labels occupy one-eighth page.
- Custom labels are placed in their defined physical dimensions.

### 11.2 Rotation

The tool should support a setting:

- Allow rotation: yes/no

If rotation is allowed, labels may be rotated 90 degrees to improve packing efficiency.

If rotation is disabled, labels keep their original orientation.

Default recommendation: rotation disabled for predictable shop use, with an option to enable it.

### 11.3 Margins and Cut-Lines in Output

The output should optionally include cut-lines around placed regions.

Settings:

- Preserve original cut-lines from source regions.
- Add new output cut-lines between mixed regions.
- Hide output cut-lines.

Default recommendation: add clean output cut-lines between mixed regions.

## 12. User Interface Requirements

The first practical version can be a small local web app or desktop-style utility.

### 12.1 Main Flow

1. User drops one or more exported PDFs into the tool.
2. Tool analyzes the PDFs.
3. Tool shows detected regions in a preview list.
4. User can confirm or adjust detections.
5. User clicks "Generate mixed A4 PDF."
6. Tool downloads the final PDF.

### 12.2 Preview Screen

For each detected source file, show:

- file name,
- page count,
- number of detected regions,
- detected region sizes,
- confidence level,
- thumbnail preview,
- manual override option.

### 12.3 Manual Override

The user must be able to manually override:

- detected split layout,
- occupied/empty region status,
- region type/size,
- rotation permission.

This is important because a wrong printout is costly and annoying in a shop workflow.

## 13. Technical Architecture

### 13.1 Recommended Prototype Stack

For a fast prototype:

- Python
- PyMuPDF / fitz for PDF rendering and page manipulation
- OpenCV for image analysis and cut-line detection
- pikepdf or PyMuPDF for PDF composition
- Streamlit or a simple local web UI for drag-and-drop usage

### 13.2 Alternative Browser-Based Stack

For a browser-only version:

- PDF.js for rendering
- canvas pixel analysis for cut-line detection
- pdf-lib for PDF composition

The browser-based version is attractive because users can run it without installing Python, but PDF composition and vector preservation may be more limited.

### 13.3 Preferred Development Path

Start with Python because it is faster for prototyping image analysis and PDF manipulation.

Once the algorithm is reliable, consider wrapping it in:

- a local web app,
- a small internal desktop utility,
- or a browser-based tool.

## 14. Algorithm Outline

For each uploaded PDF:

1. Open PDF.
2. For each page:
   1. Read page dimensions.
   2. Render page to image.
   3. Detect horizontal and vertical cut-lines.
   4. Build rectangular regions from detected cut-lines.
   5. For each region:
      1. Ignore borders and detected cut-lines.
      2. Check whether region contains meaningful content.
      3. If occupied, store crop rectangle and physical dimensions.
3. Collect all occupied regions.
4. Classify region dimensions.
5. Pack regions onto A4 output pages.
6. Copy each source region into the assigned output location.
7. Add optional output cut-lines.
8. Save final mixed PDF.

## 15. Confidence Scoring

Each detected region should receive a confidence score.

Possible confidence factors:

- cut-lines are clear and long,
- split positions match standard paper fractions,
- region dimensions match known formats,
- occupied region has enough content,
- page orientation matches expected layout,
- no ambiguous extra lines detected.

Example labels:

- High confidence
- Medium confidence
- Low confidence

If confidence is low, the tool should require user confirmation before generating.

## 16. Edge Cases

The tool should handle:

- PDFs with one full-page A4 label.
- PDFs with only one occupied slot on a multi-slot sheet.
- PDFs with multiple occupied slots.
- PDFs with no detectable cut-lines.
- PDFs with faint or broken cut-lines.
- PDFs where product content contains long black lines.
- PDFs with large white areas inside a valid label.
- PDFs with product images that are very light gray.
- Multi-page PDFs.
- Mixed portrait and landscape source pages.

## 17. Testing Plan

Use sample PDFs representing each backend output format.

Minimum test cases:

1. A4 single item.
2. A5 single item.
3. A6 single item.
4. A7 single item.
5. A5 two items on one page.
6. A6 multiple items on one page.
7. A7 multiple items on one page.
8. Shelf label.
9. Spear label.
10. In-ear label.
11. Over-ear label.
12. A realistic mixed case with 3-4 different products and formats.

For each test, verify:

- correct cut-line detection,
- correct region splitting,
- correct occupied-region detection,
- correct crop size,
- correct final scale,
- correct final output placement,
- printed result matches expected physical dimensions.

## 18. Success Criteria

The project is successful if a user can:

1. Export normal PDFs from the existing shop-admin system.
2. Drop those PDFs into the tool.
3. Generate one mixed A4 PDF.
4. Print the result at 100% scale.
5. Cut the labels using the visible cut-lines.
6. Get correctly sized labels without wasting extra sheets.

## 19. MVP Scope

The MVP should support:

- PDF upload/drop-in.
- Single-page and multi-page PDFs.
- Cut-line detection.
- Region extraction.
- Occupied-region detection.
- Basic A4 packing.
- Output PDF generation.
- Manual confirmation screen.

The MVP can initially ignore:

- perfect optimization across many pages,
- advanced rotation logic,
- fully browser-based operation,
- complex custom format classification,
- automatic integration with the shop-admin page.

## 20. Future Improvements

Possible later improvements:

- Directly intercept/download PDFs from the shop-admin page.
- Browser extension or userscript integration.
- Automatic grouping by update batch.
- Drag-and-drop manual layout editor.
- Better bin-packing optimization.
- Saved shop-specific layout presets.
- Auto-detect backend layout changes.
- Print calibration page.
- Batch mode for many files.

## 21. Open Questions

1. Should labels be allowed to rotate when packing?
2. Should the final output preserve original source cut-lines, generate new cut-lines, or both?
3. Should custom formats be packed freely or placed only in predefined safe zones?
4. Is exact physical scale more important than reducing page count in all cases?
5. Should the tool run locally, in the browser, or as an internal hosted tool?
6. Are all backend PDFs guaranteed to be A4 sheets?
7. Are cut-lines always black and straight in all formats?
8. Should the tool support multi-page source PDFs from the start?

## 22. Recommended Next Step

Build a Python proof of concept that:

1. Takes a folder of exported PDFs.
2. Detects cut-lines on each page.
3. Extracts occupied regions.
4. Outputs debug images showing detected cut-lines and crop boxes.
5. Generates one mixed A4 PDF.

Before investing in a polished interface, validate that the cut-line detection and region extraction are reliable across all known backend formats.
